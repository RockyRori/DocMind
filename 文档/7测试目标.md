## 功能需求

| 编号   | 描述                                                       |
|------|----------------------------------------------------------|
| FR01 | 用户应能够上传单个或批量pdf文件到系统                                     |
| FR02 | 系统应支持常见PDF文件格式，并能识别和拒绝非PDF文件或损坏的PDF文件                    |
| FR03 | 系统应能够从PDF文件中提取文本内容                                       |
| FR04 | 系统应能够将PDF中的图像部分分离出来                                      |
| FR05 | 系统应支持多种OCR解析模型，包括minerU、PaddleOCR、marker、mistral         |
| FR06 | 用户能够选择用于解析的OCR模型                                         |
| FR07 | 系统能够处理包含复杂布局、表格、图表和不同字体的PDF文件，并尽可能准确地提取信息                |
| FR08 | 系统应能够对从PDF中分离出的图像进行视觉分析                                  |
| FR09 | 系统应使用视觉模型（如Qwen2.5VL）自动为图像生成描述性标题                        |
| FR10 | 生成的图片标题应与图片内容高度相关且准确                                     |
| FR11 | 系统应在前端将解析后的内容渲染为Markdown格式，并支持Markdown语法（如标题、列表、表格、代码块等） |
| FR12 | 用户应能够在前端同时查看原始PDF和Markdown渲染结果                           |
| FR13 | 用户应能够下载单个或批量解析后的结果                                       |
| FR14 | 系统应支持将解析结果下载为Markdown（.md）格式                             |
| FR15 | 系统应支持将解析结果下载为DOCX（.docx）格式                               |
| FR16 | 系统应提供标准化的PDF解析API接口，允许外部系统调用                             |

## 非功能需求

| 编号    | 描述                           |
|-------|------------------------------|
| NFR01 | 单个文件上传应在N秒内完成（具体数值待定）        |
| NFR02 | 对于100页的PDF文件，解析时间应在合理范围内     |
| NFR03 | 系统应能够支持至少N个pdf 文件（具体数值待定）    |
| NFR04 | 正常的运行时间                      |
| NFR05 | 在发生故障时，系统应能够快速恢复服务，并尽量减少数据丢失 |
| NFR06 | 系统应提供清晰、友好的错误消息，指导用户解决问题     |
| NFR07 | 所有前端与后端之间的数据传输应使用HTTPS加密     |
| NFR08 | 确保不同用户上传和解析的数据相互隔离，防止数据泄露    |
| NFR09 | 系统应对所有用户输入进行严格的验证和过滤         |
| NFR10 | 代码遵循良好的编程实践和编码规范，具有高可读性和可维护性 |
| NFR11 | 前端界面应兼容主流现代浏览器               |
| NFR12 | 用户界面应直观、易于理解和操作              |
| NFR13 | 新用户应能在短时间内掌握系统的基本操作          |
